# YourKids.com — Project Overview

## What It Is

YourKids.com is a self-refreshing, AI-curated parenting content site. It combines a Python editorial engine that continuously discovers, evaluates, and writes content with an Astro static site deployed to Vercel.

The core idea: build a curated knowledge base for parents, not a news site. Content is accumulated throughout the week and published in a single weekly batch on Sunday mornings. Every article is sourced, fact-checked against its sources, and written in a consistent editorial voice defined by a living document called the Site Bible.

The site is live at https://yourkids.vercel.app/ and the code is at https://github.com/friendsip/yk.git.

## What Has Been Built

### Editorial Engine (Python)

A complete content pipeline running eight stages:

1. **Scanner** — Monitors 8 active RSS feeds (NHS, GOV.UK, NSPCC, Barnardo's, AAP, Guardian, BBC) plus 2 discovery searches. Runs every 60 minutes. Tracks consecutive failures per source and retries with backoff.

2. **Triage** — Evaluates discovered items via Claude, scoring on relevance, novelty, importance, and reliability. Runs every 30 minutes. Items scoring above 5.5 overall are passed to the planner. Low-scoring items are discarded.

3. **Planner** — Runs weekly (Sunday 05:00 UK). Receives all triaged items, the current content inventory, trend data, and recent publish history. Produces an editorial plan: which items to turn into articles, which to combine, which to hold. Outputs topic tags for each action.

4. **Writer** — Generates markdown articles from source material. Supports multi-source articles (combining several discovered items into one piece). Outputs structured YAML frontmatter with type, section, tags, and source URLs. Validates word count and required fields before staging.

5. **Editor** — Updates existing articles with new source material. Preserves voice, structure, and frontmatter. Sets `last_updated` field.

6. **Publisher** — Copies staged content to the site's content directory, commits to Git, and pushes to trigger a Vercel rebuild. Runs in auto mode (direct to main branch). Tracks all published content in the database.

7. **Synthesis** — Generates editorial overviews when opposing viewpoints are detected. Produces trend reports.

8. **Decay Checker** — Monitors published content for staleness (runs weekly, 90-day threshold). Checks for broken links and outdated information.

Supporting infrastructure:
- **Site Bible** — Living editorial document sent as context to every LLM call. Defines voice, values, guardrails, and learned context. The engine can propose amendments but cannot modify protected sections (guardrails, monetisation, GDPR).
- **SQLite database** — Tracks pipeline state, triage results, plans, published content, trends, and token usage.
- **LLM client** — All Claude API calls go through a single client that loads the Site Bible, selects the model per stage, and logs token usage for cost tracking.

**Stats:** 32 Python source files across 10 modules, 8 test files with 43 passing tests, 12 configured RSS sources.

### Static Site (Astro)

An Astro 4 site with MDX support, deployed to Vercel. No framework — plain CSS with a warm editorial design.

**Content structure:**
- Single Astro content collection (`content/`) with a `type` discriminator field
- Three content types: evergreen articles, curated links, editorials
- Five topic categories derived automatically from tags: Health, Family Life, Education, Safety, Special Needs
- Tags are kebab-case strings; display names are generated by title-casing

**Design:**
- Typography: Fraunces (display/serif) + DM Sans (body/sans-serif) via Google Fonts
- Colour palette: sage green (articles), amber (curated links), editorial purple, warm off-white background
- Responsive layout with mobile navigation
- Content cards with hover states (lift + shadow), full-card clickability via pseudo-element overlay, independently clickable tag chips

**Pages (9 templates generating 27 static pages):**
- Homepage — latest content, browse-by-topic grid, curated links section
- `/articles` + `/articles/[slug]` — evergreen articles
- `/curated` + `/curated/[slug]` — curated external links with "Read at source" button
- `/editorial` + `/editorial/[slug]` — editor's perspective pieces with editorial banner
- `/topics` + `/topics/[tag]` — tag-based category pages

**Components:** Nav, ContentCard, TypeBadge, TagList, SourcesList
**Layouts:** BaseLayout (with nav + footer), ContentLayout (shared detail page with type-specific elements)

## What Has Been Fixed

Three bugs from the first pipeline run:

1. **Duplicate plan actions** — The planner was creating one database action per source item rather than one per editorial action. Fixed by changing `plan_actions.item_id` to `item_ids` (JSON array) and creating one action per planner output.

2. **Stuck action status** — Actions that failed mid-write stayed `in_progress` forever. Fixed by adding a `staged` status after successful staging and returning stale in-progress actions (>1 hour) for retry.

3. **Silent feed failures** — RSS errors were swallowed silently. Fixed by adding consecutive failure tracking per source, retry logic in the extractor (2 retries, 30s timeout), and warnings when failures reach 3.

## Future

### Full Automation

The engine is designed to run unattended on a VPS. The remaining steps to reach full automation:

- **Deploy engine to Hetzner VPS** with systemd service management and log rotation
- **Monitoring and alerting** — track pipeline health, token costs, and publish success/failure. Alert on consecutive failures or cost spikes.
- **Automatic Site Bible evolution** — the engine already proposes amendments; adding a scheduled review and auto-approval for non-protected sections would allow the editorial voice to evolve based on what works.
- **Content decay loop** — the decay checker identifies stale content, but currently doesn't trigger rewrites. Closing this loop (decay detected → editor updates article → re-publish) would keep the entire knowledge base current without intervention.
- **Source expansion** — activating the Phase 2 RSS sources (WHO, GOV.UK Education, Ofsted, UNICEF UK) and adding more discovery searches to broaden coverage.

### Reader Interaction

The site is currently static with no user interaction. Future phases could add:

- **Newsletter** — weekly email digest of new content, built from the same publish batch. Low-complexity integration (e.g. Buttondown or Mailchimp RSS-to-email).
- **Search** — client-side search (Pagefind or Fuse.js) across all content. Straightforward to add to a static site.
- **"Was this helpful?" feedback** — lightweight per-article feedback (thumbs up/down) stored via a simple API endpoint or service like Formbricks. Feeds back into the engine's editorial planning to learn what readers value.
- **Comments or community** — longer-term consideration. A parenting site benefits from community, but moderation is non-trivial. Options range from Giscus (GitHub-backed, low moderation burden) to a dedicated community platform.
- **Personalisation** — allowing readers to follow specific topics and receive targeted content. Could be as simple as topic-based email lists or as complex as a logged-in experience with saved articles.

### Monetisation: Amazon Affiliate for Educational Toys and Books

The site already recommends books in several articles (bereavement, anxiety, therapeutic parenting). This creates a natural path to Amazon affiliate revenue without compromising editorial integrity.

**How it would work:**

- **Amazon Associates programme** — sign up for the UK programme (affiliate tag earns commission on qualifying purchases, typically 1–3% on books, up to 10% on some toy categories).
- **Curated product recommendations** — dedicated "recommended resources" sections within articles, and standalone curated lists (e.g. "Best Books for Explaining Death to Children", "Educational Toys for Sensory Play"). Each recommendation is editorially justified, not algorithmically inserted.
- **Affiliate link integration** — the engine's writer and editor stages would be updated to include Amazon affiliate links (with the associate tag appended) when referencing specific books or products. A new content type or frontmatter field (`products:`) could structure these recommendations.
- **Standalone product pages** — a `/recommended` section with curated, tagged product lists. These would be generated by the engine from a product database or maintained as a separate content collection.
- **Disclosure** — every page with affiliate links would include a clear disclosure statement, as required by Amazon's operating agreement and ASA (Advertising Standards Authority) rules. This is non-negotiable and should be built into the template.

**What to watch for:**

- **Editorial independence** — recommendations must be genuinely editorial, not driven by commission rates. The Site Bible's monetisation section (protected from automatic amendment) should codify this: recommend what's good, not what pays well.
- **Trust** — the site's value proposition is curation parents can trust. Affiliate links that feel like advertising will erode that trust quickly. The design should make clear that recommendations are editorial choices, not sponsored placements.
- **Amazon dependency** — Amazon Associates is the obvious starting point but shouldn't be the only revenue path. Bookshop.org (which supports independent bookshops) is an alternative for books. Diversifying affiliate partners reduces platform risk.
- **Content guardrails** — the existing guardrail against "advertising disguised as editorial" applies here. Product recommendations should be clearly labelled and separated from informational content. The quality test remains: "Would a knowledgeable, caring paediatrician be comfortable sharing this?"

**Implementation phases:**

1. Add affiliate links manually to existing articles that already recommend books (low effort, immediate revenue potential)
2. Build a `/recommended` section with curated product lists by topic
3. Update the engine's writer instructions to include product recommendations where editorially appropriate
4. Add a product tracking table to the database to monitor click-through and revenue by recommendation
